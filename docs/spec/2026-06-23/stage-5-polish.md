1. **마이크로인터랙션**(reanimated): 칩 체크 스케일 바운스, 완료 색 전환, 진행률 바 width 트윈, 바텀시트 드래그·딤. Stage 2에서 기본 동작만 있던 것을 PRD 6.3 기준으로 정교화한다.

4. **성능 마감**: 홈 진입 렌더·계산이 16ms/프레임 목표를 지키는지, 옵티미스틱 저장과 selector 리렌더 최소화가 유지되는지 점검한다.

## 3. 마이크로인터랙션 설계 (PRD 6.3)

Stage 2에서 칩·진행률 바·바텀시트는 **기본 동작만**(토글·표시·열림/닫힘) 구현되어 있다. Stage 5는 PRD 6.3의 네 가지 인터랙션을 reanimated로 정교화하되, 기능 동작(체크→저장→갱신)은 바꾸지 않는다(회귀 금지). 모든 애니메이션은 네이티브 스레드에서 구동해 메인 스레드 블로킹을 피한다(아키텍처 1장·8장).

### 3.1 칩 체크 스케일 바운스 (PRD 6.3)

- 칩 탭 시 `scale: 0.96 → 1.0`으로 바운스한다.
- `useSharedValue(1)` + `withSpring`(혹은 `withSequence(withTiming(0.96), withSpring(1.0))`)으로 구동. 탭 즉시 시작하고 애니메이션은 토글 상태 저장과 독립적이다(옵티미스틱 UI, PRD 6.3).
- 바운스는 시각 피드백일 뿐 완료 상태의 단일 출처가 아니다. 완료 여부는 store의 `checks` 값에서 파생한다.

### 3.2 완료 색 전환 (PRD 6.3, 6.1)

- 미완료(무채색 `--chip-idle-bg`/`--chip-idle-fg`) ↔ 완료(`--primary-weak` 배경 + `--primary` 텍스트·체크)를 **150~200ms**에 걸쳐 전환한다.
- `useAnimatedStyle` + `withTiming(target, { duration: 175 })`로 배경·텍스트 색을 보간한다. 색 토큰은 `theme/tokens.ts`만 참조한다(하드코딩 금지, 아키텍처 부록 규칙 5).
- 칩 색은 카테고리가 아니라 **완료 여부**만 표현한다(PRD 6.1). 카테고리는 라벨 텍스트로 구분한다.

### 3.3 진행률 바 width 트윈 (PRD 6.3)

- 진행률 `pct`(PRD 4.6 `weekProgress`)가 바뀌면 바의 width를 이전 값에서 새 값으로 트윈한다.
- `useAnimatedStyle`로 width를 `withTiming`한다. 값의 출처는 store selector이며, 애니메이션은 표시 계층에만 둔다(계산은 도메인 순수 함수, 아키텍처 5장).
- 주 이동(PRD 5.8)으로 다른 주의 진행률로 바뀔 때도 동일 트윈을 적용한다.

### 3.4 바텀시트 드래그·딤 (PRD 6.3, 7장)

- 상단 라운드 20px(`--radius-sheet`), 드래그 핸들, 배경 딤(어둡게). 활성 전환·공유·가져오기·운동 추가에 사용한다(아키텍처 7장).
- 드래그 제스처로 시트를 내려 닫을 수 있고, 임계치 미만 드래그는 원위치로 스냅한다(`withSpring`). 딤 불투명도는 시트 위치에 연동해 보간한다.
- Stage 2~4에서 띄우던 시트의 동작·내용은 유지하고, 드래그·딤·스냅의 부드러움만 정교화한다.

## 4. 데이터 마이그레이션 완성 (PRD 8.4)

마이그레이션 소유는 spec stage-1 §3.6에 단일 정의된다. 두 계열 중 **`migrate(raw → AppState)`(전체 상태 계열)의 변환 완성이 Stage 5의 책임**이다(공유 페이로드 계열 `migrateSharePayload`는 향후 v2 — stage-4 §6.1). Stage 1에서 `domain/migration.ts`는 **골격**(`schemaVersion` 필드 인식, `migrations[0]` v0→v1 최소 변환만 등록된 체인)만 있다. Stage 5는 이를 실제 변환 함수 체계로 채운다. 앱 시작 시 `hydrate()`(아키텍처 6장)가 `load()` 직후 마이그레이션을 호출한다(아키텍처 4장).

### 4.1 순차 변환 체인

- 저장된 `state.schemaVersion`과 앱의 `CURRENT_SCHEMA_VERSION`을 비교한다.
- 변환 함수를 버전별로 등록한다: `migrations[n]`은 `schemaVersion n → n+1` 변환을 담당한다.
- 저장 버전이 현재보다 낮으면 `n`부터 `current-1`까지 **순차 적용**한다. 같으면 통과, 높으면(앱이 구버전) 명확한 안내 후 거부한다(PRD 7.2 호환성 정책과 정합).

```typescript
// domain/migration.ts (구조 개요)
const migrations: Record<number, (s: unknown) => unknown> = {
  // 1: (s) => ({ ...s, schemaVersion: 2, /* v1→v2 변환 */ }),
}

function migrate(raw: unknown): AppState {
  let s = raw as { schemaVersion: number }
  if (s.schemaVersion > CURRENT_SCHEMA_VERSION) {
    throw new IncompatibleVersionError(s.schemaVersion) // 앱 업데이트 필요
  }
  while (s.schemaVersion < CURRENT_SCHEMA_VERSION) {
    s = migrations[s.schemaVersion](s) as typeof s      // n → n+1
  }
  return AppStateSchema.parse(s) // Zod로 변환 결과 무결성 검증(아키텍처 1장)
}
```

### 4.2 마이그레이션 전 자동 백업·실패 시 롤백 (PRD 8.4, 10.2 리스크)

- 변환을 시작하기 전에 원본(변환 전 raw)을 별도 백업 키로 보존한다(예: `appstate.backup.preMigration`).
- 변환 체인 또는 Zod 검증이 실패하면 백업으로 **롤백**하고, 마이그레이션을 적용한 저장은 하지 않는다. 사용자에게 실패를 알리고 데이터 손실을 막는다(PRD 10.2 "마이그레이션 실패로 데이터 손실" 완화).
- 변환이 모두 성공하면 새 `AppState`를 저장하고, preMigration 백업 정리 시점 등 상세는 `docs/development/2026-06-23/stage-5-polish.md`에서 정한다.

### 4.3 버전 불변성과 versionId 참조 보존 (PRD 8.4, 4.1)

두 개념을 분리해 서술한다. 표면상 충돌처럼 보이지만 적용 경로가 다르다.

- **버전 append-only 불변성**(PRD 4.1)은 **런타임 앱 편집** 불변식이다. 사용자가 루틴을 편집하면 옛 `RoutineVersion`을 변형하지 않고 새 버전을 append한다.
- **마이그레이션**(PRD 8.4)은 **`schemaVersion` 진화** 경로다. 저장된 `AppState` 전체를 새 스키마 구조로 변환하는 별개 작업이며, 런타임 편집 불변식과는 다른 차원이다.

핵심 보증: **변환 후에도 모든 `DayLog.versionId` 참조가 실재 `RoutineVersion`으로 유효하게 해소된다**(PRD 8.4). 계획 해소의 단일 출처는 활성 타임라인이므로(PRD 4.4 필드 설명), 마이그레이션은 타임라인 엔트리의 `versionId`와 `DayLog.versionId`가 가리키는 버전이 변환 후에도 존재하도록 보장해야 한다.

- 대부분의 마이그레이션은 필드 추가·변형이라 `versionId` 값이 그대로 유지되어 참조가 자동으로 유효하다.
- 단, **`versionId`를 재키잉(rekey)하는 마이그레이션이면 같은 변환 패스 안에서 `activationTimeline`과 `DayLog`의 `versionId` 참조도 함께 remap**해야 한다. 변환은 (옛 versionId → 새 versionId) 매핑을 만들고, 버전 객체·타임라인·완료 로그 세 곳에 일관 적용한다(PRD 8.4 "버전 객체 구조를 바꾸면 참조 매핑을 함께 갱신").

## 5. 로컬 백업/복원 (PRD 8.5)

설정 탭 화면(`(tabs)/settings.tsx`)은 Stage 5에서 처음 스캐폴딩되며, 백업/복원과 초기화 진입점을 담는 그릇이다. 이 화면에서 전체 `AppState`를 JSON으로 내보내고 가져온다. 7장 공유와 목적·범위가 다르다.

| 구분 | 백업/복원 (8.5, 이 절) | 공유/임포트 (7장, Stage 4) |
|------|------------------------|-----------------------------|
| 페이로드 범위 | **전체 `AppState`**(루틴·타임라인·**완료 로그 포함**·설정) | **루틴 템플릿만**(완료 로그·내부 id·타임라인 제외, D8.5) |
| 목적 | v1 동기화 부재 보완용 수동 전체 백업(PRD 8.5) | 1:1 루틴 공유(PRD D7) |
| id/slotId | 그대로 보존(자기 데이터 복원) | 임포트 시 재발급(충돌 회피) |
| 직렬화 | 도메인 직렬화 재사용(아키텍처 4장 주석) | 동일 도메인 직렬화 재사용 |

### 5.1 내보내기

- store의 현재 `AppState`를 그대로 JSON 직렬화한다(`schemaVersion` 포함). 공유와 달리 완료 로그·id·타임라인을 모두 담는다.
- expo 파일 시스템에 파일로 쓰고 공유 시트로 내보낸다. 파일명에 날짜·`schemaVersion`을 포함해 식별을 돕는다.

### 5.2 가져오기 (검증·확인 단계)

복원은 자기 데이터를 통째로 덮어쓰므로 검증·확인을 거친다.

1. 파일 선택 → JSON 파싱.
2. **Zod 스키마 검증**(아키텍처 1장)으로 구조 무결성을 확인한다. 깨진 파일은 거부한다.
3. `schemaVersion` 확인: 현재보다 낮으면 4장 마이그레이션 체인을 적용하고, 높으면 거부한다(PRD 7.2와 동일 정책).
4. 복원이 기존 데이터를 덮어씀을 사용자에게 명확히 고지하고 확인을 받는다(실수 방지, PRD 5.10.2 초기화 확인 패턴과 동질).
5. 확인 시 전체 `AppState`를 교체하고 Repository로 저장한다.

## 6. 성능 마감 (PRD 8.3)

이 절은 **점검**이지 신규 최적화·프로파일링 인프라 구축이 아니다. PRD 8.3은 스트릭 60일 소급을 "단순 루프이므로 부담 없음"으로 못 박았다. 새 구조를 만들지 않고 아래를 확인한다.

- **홈 진입 16ms/프레임 점검**(PRD 8.3): 이번 주 렌더링과 `weekProgress`·`streak` 계산이 메인 스레드를 16ms 이상 막지 않는지 점검한다. 도메인 함수는 순수·무상태이므로(아키텍처 5장) 입력만 주어 단독 측정 가능하다.
- **옵티미스틱 저장 확인**(PRD 6.3·8.3): 칩 토글이 store를 즉시 갱신해 UI에 반영하고, `Repository.save`는 비동기로 수행하며, 실패 시에만 토스트로 알리는 동작이 유지되는지 확인한다(아키텍처 6장).
- **selector 리렌더 최소화 확인**(아키텍처 6장): 홈이 "이번 주 진행률·스트릭·요일별 칩 상태"만 좁게 구독해, 잦은 토글이 무관한 화면·컴포넌트를 리렌더하지 않는지 확인한다.

## 7. 기술 노트

- **reanimated 네이티브 스레드 구동**: 칩 바운스·색 전환·진행률 트윈·시트 드래그는 `useSharedValue`/`useAnimatedStyle`/`withTiming`/`withSpring`/제스처로 UI 스레드에서 구동한다(아키텍처 1장·8장). JS 스레드의 store 갱신과 분리해 옵티미스틱 UI의 즉각성을 유지한다.
- **백업 파일 입출력**: expo 파일 시스템으로 JSON 파일을 쓰고, 공유 시트로 내보낸다. 가져오기는 파일 선택 → 읽기 → 파싱 경로다. 구체 모듈·API는 설치 시점 Expo SDK에 맞춰 `docs/development/2026-06-23/stage-5-polish.md`에서 확정한다(아키텍처 1장 비고).
- **마이그레이션 테스트용 구버전 fixture**: `__tests__`에 과거 `schemaVersion` 형태의 `AppState` 고정 데이터를 둔다. fixture를 이용한 변환 체인 왕복·참조 보존 검증은 `docs/development/2026-06-23/stage-5-polish.md` 참조.
- **레이어 경계**: 마이그레이션·직렬화·계산은 모두 `domain/`(순수 함수), 저장·파일 입출력은 Repository·플랫폼 레이어가 담당한다. UI는 표시·트리거만 한다(아키텍처 2장 의존 방향).

> 상세 테스트 케이스·종료 조건·출시 전 통합 점검 체크리스트는 `docs/development/2026-06-23/stage-5-polish.md` 참조.
