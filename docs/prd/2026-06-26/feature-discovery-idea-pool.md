# 운동 관리앱 경쟁 기능 발굴 — 아이디어 풀

- **작성일**: 2026-06-26
- **제품 단계**: 기존 제품 (dailyroutine v1, M1~M5 구현 완료 · 출시 전 · 단일 사용자 개인 앱)
- **발굴 범위**: v1.x 로컬 우선 (백엔드·계정 없이 단일 기기에서 더할 수 있는 것 중심)
- **발굴 목적**: 아이디어 풀 확보 (넓게 발굴·정리. 우선순위·실험 설계는 가볍게)
- **발굴 질문**: "우리가 만든 루틴 체크 + 스트릭 + 공유 모델을 기준으로, 다른 운동 관리앱들이 흔히 제공하는 것 중 우리에게 더 필요한 기능은 무엇인가?"

> 본 문서는 발산(divergent) 단계의 아이디어 풀이다. 각 아이디어는 "대표 앱들이 흔히 제공하는 패턴"에서 착안했으며, 특정 앱의 정확한 기능 사양을 단정하지 않는다. 채택·우선순위·구현 결정은 별도 PRD/설계 단계에서 수행한다.

## 0. 발굴 맥락

### 우리 앱이 이미 가진 것 (v1)

- 주간 운동 루틴 생성·편집 (요일 고정, 세트 정보는 자유 문자열 예: `"4 × 12"`)
- 매일 체크오프 (`completionLogs`)
- 주간 진행률 (`weekProgress`: 활동일×슬롯 보유 카테고리 기준 done/total/pct)
- 연속 달성 스트릭 (`streak`: 휴식일·미활성일·빈 활동일은 면제, 오늘 미완은 끊지 않음)
- 루틴 공유: 코드 문자열 / 딥링크 / QR (export·import)
- 전체 백업·복원 (로컬 JSON 파일)
- 단일 라이트 테마, 로컬 저장(AsyncStorage), 월요일 시작 주, 기기 로컬 타임존

### 벤치마크 대상 (대표 앱 전반)

| 앱 | 성격 | 우리가 착안할 영역 |
|----|------|----------|
| Strong · Hevy · Jefit | 근력·루틴 트래커 | 운동 라이브러리, PR·기록, 휴식 타이머, 노트, 루틴 템플릿 |
| Strava | 러닝·소셜 | 활동 히트맵·캘린더, 챌린지, 배지 |
| Nike Training Club | 가이드 트레이닝 | 추천 루틴·온보딩 마찰 감소 |
| Fitbod | 적응형 프로그래밍 | 데이터 기반 인사이트 |
| Duolingo | 스트릭·게임화 | 스트릭 보호, 마일스톤 축하, 리마인더, 위젯 |

### 발굴 렌즈 3종

- **PM 렌즈**: 가치 — 사용자가 계속 쓰게 만드는가 (리텐션·동기)
- **Designer 렌즈**: 사용성 — 매일 쓰기 쉽고 보람 있는가
- **Engineer 렌즈**: 실현성 — 우리 데이터 모델·로컬 제약 안에서 만들 수 있는가

> 개인·비상업 앱이므로 통상 발굴의 "수익성·시장 진입(GTM)" 잣대는 적용하지 않는다.

## 1. 우리 앱 vs 경쟁 앱 — 기능 격차 한눈에

| 영역 | 우리 v1 | 대표 앱들이 흔히 제공 | 격차 |
|------|---------|----------------------|------|
| 일일 수행 추적 | 체크오프 | 체크 + 세트·무게·반복 로깅 | 우리는 자유 문자열까지만 (v1 의도) |
| 진행 동기부여 | 주간 진행률 + 스트릭 | + 배지·마일스톤·축하 연출 | 보상 피드백 약함 |
| 장기 추세 시각화 | 없음 (이번 주 한정) | 활동 캘린더·히트맵·통계 | **데이터는 있으나 시각화 없음** |
| 지속 유도 | 없음 | 리마인더 알림 | 망각 시 복귀 트리거 없음 |
| 스트릭 회복력 | 활동일 누락 시 끊김 | 스트릭 프리즈·면제 | 아픔·여행에 취약 |
| 온보딩 | 빈 루틴부터 | 추천 루틴·템플릿 라이브러리 | 시작 마찰 |
| 운동 중 보조 | 없음 | 휴식 타이머·노트 | 세션 중 쓸모 약함 |

핵심 관찰: **우리는 이미 풍부한 `completionLogs`를 쌓고 있으나, 그것을 동기로 되돌려주는 출구(시각화·보상·복귀 트리거)가 거의 없다.** 가장 가성비 높은 발굴 영역이다.

## 2. 아이디어 풀 (3 렌즈 · 13종)

각 항목 표기: **착안 앱** / **무엇** / **우리 체크오프 모델에 맞는 이유** / **범위 태그** / 가치·사용성·실현성 한 줄.

범위 태그 정의:
- `[v1.x 로컬]` 백엔드 없이 당장 가능
- `[PRD 재검토 플래그]` 기술적으로 v1.x 가능하나 PRD가 다른 범위로 분류해 둠 → 무단 승격하지 않고 결정 요청
- `[v2·참고]` 백엔드·대규모 네이티브 작업 필요. 참고용

### Lens A — PM (가치 · 리텐션)

**A1. 운동 히스토리 캘린더 · 잔디 히트맵** `[v1.x 로컬]` ★최우선급
- 착안: Strava 활동 캘린더, GitHub 잔디
- 무엇: 월간 캘린더·연간 히트맵으로 완료 이력 시각화
- 맞는 이유: `completionLogs`에 데이터가 이미 있다. 집계·렌더만 추가하면 됨. 신규 저장 모델 불필요
- 가치 높음 · 사용성 높음 · 실현성 높음 (데이터 재활용, react-native-svg 보유)

**A2. 스트릭 보호 (프리즈/면제 토큰)** `[v1.x 로컬]` · `[PRD 4.7 영향]`
- 착안: Duolingo streak freeze
- 무엇: 예정된 운동일을 못 지킨 날(아픔·여행)에도 스트릭을 1~2회 지켜주는 보호 장치
- 맞는 이유: **결함 보정이 아니다.** `streak()`는 이미 휴식일·미활성일을 면제한다. 끊기는 경우는 *예정된 활동일을 미수행·미기록*했을 때뿐. 그 현실적 누락에 대한 보험
- 주의: 스트릭 정의(PRD 4.7) 변경이므로 단순 추가 아님. 보호 토큰 수·획득 방식 설계 필요
- 가치 중간~높음 · 사용성 중간 · 실현성 중간

**A3. 행동 마일스톤 배지** `[v1.x 로컬]`
- 착안: Duolingo 마일스톤, Strava 업적
- 무엇: "최장 스트릭 14일", "총 완료 100회", "4주 연속 주간 목표 달성" 같은 행동 기반 배지
- 맞는 이유: 무게 PR은 자유 문자열이라 어렵지만, **행동 지표(횟수·연속)는 우리 데이터로 즉시 산출** 가능
- 가치 중간 · 사용성 높음 · 실현성 높음

**A4. 주 N회 유연 목표 모드** `[v1.x 로컬]` · `[모델 변경 큼]`
- 착안: 다수 앱의 "주당 횟수" 목표
- 무엇: 요일 고정 대신 "이번 주 4회 중 2회" 같은 유연 목표
- 맞는 이유: 요일 고정이 안 맞는 사용자 수용. 단 `weekProgress`/`streak` 분모 모델을 바꿔야 함 (활동일×카테고리 → 횟수 기반). 큰 변경
- 가치 중간 · 사용성 중간 · 실현성 중간~낮음 (모델 재설계)

### Lens B — Designer (사용성 · 동기)

**B1. 로컬 리마인더 알림** `[PRD 재검토 플래그]` ★
- 착안: 거의 모든 앱의 핵심 리텐션 도구
- 무엇: "오늘 운동 체크를 안 했어요" 정시 로컬 알림
- 맞는 이유: **`expo-notifications` 로컬 스케줄 알림은 백엔드가 전혀 필요 없다** — 기술적으로 v1.x 로컬 가능
- 플래그: 다만 PRD 9.2가 "푸시 알림"을 v2/향후로 분류해 둠. *로컬* 알림은 서버 불필요라는 점을 들어 **범위 재검토를 요청**하는 것이지, 무단 v1.x 승격이 아님. 범위 결정 필요
- 가치 높음 · 사용성 높음 · 실현성 높음 (단 범위 결정 선행)

**B2. 항목별 메모 · 컨디션 노트** `[v1.x 로컬]`
- 착안: Hevy/Strong 운동 노트
- 무엇: 각 체크 항목·날짜에 짧은 메모(무게·컨디션·느낌)
- 맞는 이유: 자유 문자열 세트 모델과 결이 같음. 스키마에 옵션 필드 추가 + 마이그레이션
- 가치 중간 · 사용성 중간 · 실현성 중간

**B3. 주간 리뷰 요약 카드 (텍스트)** `[v1.x 로컬]` · `[그래프는 v2]`
- 착안: 다수 앱의 주간 회고
- 무엇: "이번 주 8개 중 5개 완료(63%), 스트릭 12일, 가장 잘 지킨 운동은 X" 텍스트 요약 카드
- 맞는 이유: 기존 `weekProgress`/`streak` + 간단 집계로 구성. **그래프 차트는 PRD가 향후로 분류했으나 텍스트 요약은 경량** → 그래프 없이 요약만 v1.x
- 가치 중간 · 사용성 중간~높음 · 실현성 높음

**B4. 휴식 타이머** `[v1.x 로컬]`
- 착안: Strong/Hevy 핵심
- 무엇: 세트 간 휴식 카운트다운
- 맞는 이유: 운동 중 보조. 단 우리는 "체크" 단위라 적합도 중간(세트 단위 추적이 없음). 독립 유틸로 붙일 수 있음
- 가치 중간 · 사용성 중간 · 실현성 높음

### Lens C — Engineer (실현성 · 데이터)

**C1. 내장 루틴 템플릿 라이브러리** `[v1.x 로컬]` ★
- 착안: Jefit/NTC 추천 프로그램
- 무엇: "초보 3분할", "주 3회 전신" 같은 미리 만든 루틴을 앱 내장 JSON으로 제공 → 한 번에 가져오기
- 맞는 이유: **서버 불필요(앱 번들 JSON)**. 우리 공유/임포트 파이프라인을 그대로 재사용. 빈 루틴 시작 마찰을 크게 줄임
- 가치 높음 · 사용성 높음 · 실현성 높음 (기존 import 코드 재활용)

**C2. 데이터 인사이트 · 통계** `[v1.x 로컬]`
- 착안: Fitbod/Strava 통계
- 무엇: 완료율 추세, 요일별 수행률, 가장 잘 지킨/놓친 운동
- 맞는 이유: `completionLogs` 순수 집계. 도메인 함수로 테스트 용이
- 가치 중간 · 사용성 중간 · 실현성 높음

**C3. 과거 날짜 소급 체크 · 보정** `[v1.x 로컬]` · `[불변식 충돌 플래그]`
- 착안: 일부 트래커의 과거 기록 보정
- 무엇: 깜빡한 날을 나중에 소급 체크
- 맞는 이유 & 긴장: 사용성엔 좋으나 **"오늘 보호" 불변식(D8.8)과 정면 충돌**한다. PRD 10.2가 1순위 리스크로 지목한 영역. 자유로운 추가가 아니라, 불변식을 어디까지 완화할지 정책 결정이 선행되어야 함 — 긴장 유지
- 가치 중간 · 사용성 중간 · 실현성 낮음 (불변식 재설계 위험)

**C4. 홈/잠금화면 위젯 (스트릭·진행률)** `[v2·참고]`
- 착안: Duolingo/Strava 위젯
- 무엇: iOS 위젯에 오늘 진행률·스트릭 표시
- 맞는 이유 & 한계: 동기부여 효과 크나 네이티브 위젯 작업이 무겁고 PRD가 향후로 분류. 참고로만
- 가치 높음 · 사용성 높음 · 실현성 낮음 (네이티브 비용)

**C5. 소셜 · 친구 챌린지** `[v2·참고]`
- 착안: Strava/Hevy 소셜 피드
- 무엇: 친구와 스트릭·완료 경쟁, 응원
- 맞는 이유 & 한계: 백엔드·계정 필수 → v1 범위 밖. PRD 7.3 v2 로드맵과 정합. 참고로만
- 가치 중간(개인 앱 특성상) · 실현성 낮음 (백엔드 필요)

## 4. 상위 후보 가벼운 검증 메모 (Top 5)

아이디어 풀 목적이므로 실험 설계 대신, 각 후보의 *가장 불확실한 가정*과 *한 줄 검증법*만 남긴다.

| # | 아이디어 | 가장 불확실한 가정 | 한 줄 검증법 |
|---|----------|-------------------|-------------|
| 1 | A1 히스토리 히트맵 | "장기 시각화가 실제로 지속 동기를 높이는가" (가치) | 한 화면 프로토타입을 2주 직접 써보고 체크 빈도 변화 관찰 |
| 2 | C1 루틴 템플릿 | "빈 루틴 마찰이 실제 시작 장벽인가" (가치) | 템플릿 3종 임포트 후, 직접 만든 루틴 대비 유지율 비교 |
| 3 | B1 로컬 알림 | "알림이 거슬림 없이 복귀를 유도하는가" (사용성) + PRD 범위 결정 (선행) | 먼저 범위 결정 → 1주 알림 on, 무시율·체크 복귀율 측정 |
| 4 | A2 스트릭 보호 | "보호가 동기를 지키는가, 아니면 스트릭 의미를 희석하는가" (가치) | 보호 토큰 규칙안 2개를 종이 시뮬레이션으로 비교 |
| 5 | A3 마일스톤 배지 | "행동 배지가 보상으로 느껴지는가" (가치) | 배지 카피·달성 시점 시안을 만들어 체감 반응 점검 |
