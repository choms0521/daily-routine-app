# 운동 트래커 PRD (제품 요구사항 정의서)

> 문서 버전: v1.2
> 작성일: 2026-06-22
> 상태: 승인 (독립 검토 APPROVE, single source of truth)
> 산출물 범위: 제품 요구사항 정의. 상세 화면 설계와 구현은 별도 문서로 파생한다.

---

## 0. 이 문서를 읽는 법

이 PRD는 개인용 운동 트래커 앱의 단일 출처(single source of truth)다. 이후의 상세 설계, 데이터 스키마 확정, 화면 설계, 구현은 모두 이 문서의 결정을 따른다.

핵심 한 줄: **루틴 정의(definition)와 완료 기록(completion log)을 분리하고, 루틴은 버전으로 관리하며, 과거 기록은 그날 활성이던 버전으로 재계산한다.** 이 결정이 데이터 모델, 스트릭, 진행률, 공유 설계의 뿌리이며, 다른 모든 것은 이 결정의 하위다. 문서 전체는 이 제약을 위반하지 않도록 작성되었다.

## 1. 개요

### 1.1 한 줄 정의

사용자가 직접 만든 주간 운동 루틴을 매일 체크하고, 주간 진행률과 연속 달성(스트릭)을 추적하며, 루틴을 다른 사람과 코드/링크/QR로 주고받을 수 있는 1인용 모바일 운동 트래커.

### 1.2 문제 정의

기존 프로토타입은 "아침 러닝 + 저녁 근력"이라는 특정 스케줄을 코드에 하드코딩했다. 이 때문에:

- 다른 종류의 운동(수영, 사이클, 요가, 크로스핏 등)을 추적할 수 없다.
- 스케줄을 바꾸려면 코드를 고쳐야 한다.
- 자신의 루틴을 다른 사람과 공유할 방법이 없다.
- 운동 분류가 "러닝/근력" 2종에 고정되어 일반성이 없다.

사용자는 **자기만의 루틴을 자유롭게 만들고, 바꾸고, 공유하면서도**, 과거의 진행률과 스트릭이 정확하게 유지되기를 원한다. 또한 기본 운동 종류(러닝, 수영, 사이클, 요가, 스쿼트 등)를 미리 목록으로 제공받아 빠르게 시작하되, 목록에 없는 운동은 직접 커스텀으로 추가할 수 있기를 원한다.

### 1.3 목표 (Goals)

| # | 목표 | 측정 지표 |
|---|------|----------|
| G1 | 사용자가 코드 수정 없이 루틴을 완전히 커스터마이즈할 수 있다 | 루틴 에디터로 운동·세트·요일 배정·휴식일을 100% GUI로 편집 가능 |
| G2 | 매일의 체크가 즉각적이고 마찰 없다 | 칩 탭 → 저장까지 사용자 추가 동작 0회 |
| G3 | 루틴을 편집·공유해도 과거 진행률·스트릭이 변하지 않는다 | 루틴 편집 전후 과거 주(week)의 진행률·스트릭 값이 불변 |
| G4 | 서버 비용 없이 루틴을 공유할 수 있다 | v1 공유는 코드/딥링크/QR만으로 동작, 백엔드 호출 0건 |
| G5 | 일관된 Toss 라이트 디자인 언어를 따른다 | 화이트 배경 + Toss Blue 단일 주 색상, 숫자 중심 UI |

### 1.4 비목표 (Non-goals, v1 기준)

- **멀티 기기 동기화**: v1은 단일 기기 로컬 저장만. 동기화는 v2 백엔드 승격 시 도입.
- **공개 루틴 갤러리/검색**: v1은 1:1 공유(코드/링크/QR)만. 갤러리는 v2.
- **소셜 기능**: 팔로우, 좋아요, 댓글 등은 범위 밖.
- **러닝 거리·페이스 자동 측정**: HealthKit/Google Fit 연동은 향후 확장(11장).
- **세트별 무게·반복수 로깅**: 점진적 과부하 추적은 향후 확장(11장). v1은 "체크 완료" 단위까지만.
- **계정/로그인**: v1은 계정 없음. 모든 데이터는 기기 로컬.
- **다국어**: v1은 한국어만.

## 2. 타깃 사용자 & 핵심 시나리오

### 2.1 페르소나

**"혼자 운동하는 사람" (1인 사용자)**

- 규칙적인 주간 운동 루틴을 가지고 있다(예: 주 5회 유산소 + 주 4회 무산소, 일요일 휴식).
- 운동 후 "오늘 거 했다"를 빠르게 기록하고 싶다.
- 며칠 연속으로 해냈는지(스트릭)가 동기 부여가 된다.
- 가끔 루틴을 바꾼다(계절/목표/부상에 따라).
- 친구가 좋은 루틴을 만들면 그대로 받아서 쓰고 싶다.
- 서버에 자기 데이터가 올라가는 것을 원하지 않는다(프라이버시).

### 2.2 핵심 유저 저니

**저니 A — 오늘 체크 (가장 빈번, 하루 1~3회)**

1. 앱을 연다 → 홈에 이번 주가 보이고, 오늘 요일이 강조되어 있다.
2. 오늘 카드의 유산소 칩을 탭한다 → 즉시 완료 표시 + 저장.
3. 무산소 칩 옆 펼치기를 눌러 운동 목록을 본다.
4. 운동을 하나씩 체크하거나, 무산소 칩을 직접 탭해 전부 일괄 체크한다.
5. 상단 진행률 바와 스트릭 숫자가 즉시 갱신된다.

**저니 B — 루틴 생성·편집 (가끔)**

1. 라이브러리에서 "새 루틴 만들기"를 누른다.
2. 루틴 이름을 정하고, 유산소/무산소 카테고리에 운동을 추가한다(세트 정보 포함).
3. 각 운동을 요일에 배정하고, 휴식일을 지정한다.
4. 저장 → 새 루틴이 라이브러리에 생긴다. "활성으로 설정"할 때:
   - **이전에 활성 루틴이 없었다면**(최초 활성화) 오늘부터 홈에 즉시 반영된다.
   - **이미 다른 루틴이 활성이었다면**(전환) 새 루틴을 적용하기 전에 "기존 활성 루틴이 꺼지고 새 루틴으로 전환됩니다"라는 안내를 팝업으로 명확히 표시하고, 사용자가 동의한 경우에만 전환을 진행한다. 동의하면 오늘은 기존 루틴이 그대로 유지되고, **내일부터** 새 루틴이 홈에 반영된다(오늘 보호, D8.8). 홈에 "내일부터 적용" 안내를 표시한다.
5. 나중에 루틴을 편집하면 **새 버전이 생성**되고, **내일부터의 계획만** 바뀐다. 오늘과 과거 기록은 그대로다(오늘 보호, D8.8).

**저니 C — 루틴 공유받아 사용 (가끔)**

1. 친구가 보낸 공유 코드(또는 딥링크/QR)를 받는다.
2. 앱에서 "루틴 가져오기" → 코드 붙여넣기 / 링크 탭 / QR 스캔.
3. 미리보기로 루틴 내용을 확인한다(운동·세트·요일·휴식일).
4. "내 라이브러리에 추가"를 누른다 → 라이브러리에 새 루틴으로 들어온다.
5. 원하면 활성으로 전환해 바로 사용한다. 가져온 루틴은 내 것처럼 편집할 수 있다(편집 시 새 버전 생성).

## 3. 제품 결정 기록 (의사결정 로그)

각 결정은 "결정 / 근거 / 대안 / 트레이드오프" 형식으로 기록한다. D1~D7은 사용자 재가 완료(재논의 금지), D8은 본 PRD의 핵심 아키텍처 제약이다.

### D1. 산출물 = PRD 마크다운 1편

- **결정**: 이 단계의 산출물은 PRD 마크다운 문서 1편이다.
- **근거**: 구현 이전에 단일 출처를 확정해야 후속 설계·구현이 일관된다.
- **대안**: 곧바로 구현 시작 / 디자인 시안 먼저.
- **트레이드오프**: 문서 작성 시간이 들지만, 내부 모순(특히 버전 관리 누락)으로 인한 재작업 비용을 막는다.

### D2. 플랫폼 = React Native + Expo (iOS/Android)

- **결정**: React Native + Expo로 iOS/Android 동시 타깃. 저장소는 AsyncStorage를 v1 기본으로 권고(상세 8장).
- **근거**: 사용자가 RN/Expo 경험 보유. Expo Go로 빠른 개발·배포. 단일 코드베이스로 양 플랫폼 커버.
- **대안**: Next.js + PWA / 네이티브 분리 개발.
- **트레이드오프**: MMKV 같은 고성능 저장소는 Expo Go에서 동작하지 않아 dev build가 필요(8장 참조). v1 데이터 규모상 AsyncStorage로 충분하다.

### D3. 루틴 범위 = 풀 커스텀 에디터 + 공유/임포트

- **결정**: 사용자가 루틴/스케줄/운동/세트를 자유롭게 생성·편집한다. 루틴 공유/임포트 기능을 포함한다.
- **근거**: 하드코딩 제거가 일반화의 핵심 가치. 공유는 1인 앱에 네트워크 효과를 더한다.
- **대안**: 프리셋 몇 개만 제공 / 공유 없음.
- **트레이드오프**: 에디터 UX 복잡도와 버전 관리 비용이 증가하지만, 이것이 제품의 핵심 차별점이다.

### D4. 운동 분류 = 유산소 / 무산소 2대 카테고리

- **결정**: 운동을 유산소(aerobic) / 무산소(anaerobic) 2개 카테고리로 일반화한다. 기존 "러닝"은 유산소로, "근력"은 무산소로 흡수한다. 각 카테고리 안에 여러 운동 종목이 있고, 종목은 세트 정보를 가진다.
- **근거**: 2대 분류는 운동 과학의 보편적 구분이며 거의 모든 종목을 수용한다. 기존 러닝/근력 2종을 유산소/무산소 카테고리로 자연 승계할 수 있다. (단 시각 구분은 기존의 색상 코드가 아니라 칩 라벨 텍스트로 한다 — 칩 색은 완료 여부만 표현, 6.1.)
- **대안**: 자유 태그 분류 / 부위별 분류(상체·하체·코어) / 단일 분류.
- **트레이드오프**: 일부 복합 운동(크로스핏 등)은 한 카테고리에 강제 배정되지만, 2분류의 단순함이 일관된 완료 규칙·진행률 계산을 가능하게 한다.
- **세트 정보 모델(확정)**: v1은 세트 정보를 **자유 문자열**(`"4 × 12"`, `"3 × 40초"`)로 저장한다. 구조화(무게·반복수 분리 필드)는 점진적 과부하 로깅(11장)이 필요한 v2에서 도입하며, 그때 문자열 → 구조 마이그레이션을 수행한다. v1에서 자유 문자열을 택한 이유: 입력 마찰 최소화, 다양한 표기(시간/거리/세트×반복) 수용.

### D5. 디자인 = 풀 Toss 라이트

- **결정**: 화이트 배경 + Toss Blue(#3182F6) 단일 주 색상. 숫자 중심(number-forward), 바텀시트, 단일 주 CTA, 마이크로인터랙션, 넉넉한 여백 등 Toss 디자인/인터랙션 언어를 차용한다. 기존 다크 토큰(#15171C 등)은 폐기한다.
- **근거**: 라이트 + 단일 주 색상은 숫자·진행률·스트릭 같은 핵심 정보를 명료하게 드러낸다. 일관된 디자인 시스템이 화면 추가 시 drift를 막는다.
- **대안**: 다크 유지 / 멀티 컬러 / 머티리얼 디자인.
- **트레이드오프**: 다크 모드 선호 사용자는 향후 확장 필요. v1은 단일 라이트 테마로 단순성 확보.

### D6. 루틴 모델 = 루틴 라이브러리 (다수 보유, 1개 활성)

- **결정**: 사용자는 여러 루틴을 라이브러리에 보유하고, 한 번에 1개만 "활성" 상태다. 활성 루틴을 전환할 수 있다. 공유받은 루틴을 라이브러리에 추가한다.
- **근거**: 계절/목표별 여러 루틴을 운영하는 현실을 반영. "활성 1개" 제약이 홈 화면과 진행률·스트릭 계산을 단순화한다.
- **대안**: 단일 루틴만 / 여러 루틴 동시 활성.
- **트레이드오프**: "동시 활성" 유연성을 포기하지만, 활성 전환 이벤트를 timeline에 기록하면 과거 재계산이 명확해진다(D8).

### D7. 공유 아키텍처 = 서버리스 우선 → 백엔드 단계적 승격

- **결정**: v1은 루틴 템플릿을 JSON으로 직렬화하여 **공유 코드 문자열 / 딥링크 / QR**로 전달·임포트한다(서버 비용 0). v2에서 Supabase 등 경량 백엔드로 공유 링크/공개 갤러리/멀티기기 동기화를 승격한다.
- **근거**: 1인 앱에서 초기 서버 비용·운영 부담을 0으로 유지. 직렬화 공유는 즉시 동작하고 프라이버시 친화적.
- **대안**: 처음부터 백엔드 / 공유 없음.
- **트레이드오프**: v1 공유 코드는 길이 제한·수동 전달의 불편이 있으나, 압축·딥링크·QR로 완화하고 v2에서 정식 링크로 승격한다.
- **세부 정책(확정)**: 페이로드를 압축 + URL-safe Base64로 최소화한다. QR 단일 프레임 용량을 초과하면 QR을 비활성화하고 코드 문자열 공유로 폴백한다. **멀티 프레임 QR은 v1 범위에서 제외**한다(7.1). 큰 루틴의 근본적 길이 문제는 v2 링크 단축으로 해결한다.

### D8. (핵심 제약) 루틴 정의와 완료 기록 분리 + 버전 관리

- **결정**:
  1. 완료 기록(completion log)은 루틴의 **버전/스냅샷을 참조**한다.
  2. 과거 주는 **각 날짜에 활성이던 버전**으로 재계산한다.
  3. 루틴 편집은 옛 버전을 변형(mutate)하지 않고 **새 버전을 생성**한다.
  4. 공유 루틴 임포트 = 템플릿 버전 채택.
  5. 공유 페이로드 = **루틴 템플릿만** 담는다(완료 로그 제외) → 작고 서버리스 친화적.
  6. "휴식일"은 캘린더가 아니라 **루틴의 데이터 속성**이다.
  7. 파생값 "카테고리 완료 = 해당 요일 항목 전부 체크"는 일반화하여 보존한다.
  8. **`effectiveFrom` 정책 (오늘 보호 불변식)**: 타임라인 엔트리의 적용 시작일을 다음으로 고정한다.
     - 최초 활성화(이전 활성 루틴이 없던 상태 → 첫 활성): `effectiveFrom = 오늘`. (보호할 오늘 기록이 없으므로 당일부터 적용.)
     - 활성 루틴 **편집**: `effectiveFrom = 오늘+1(내일)`.
     - 활성 루틴 **전환**(A→B): `effectiveFrom = 오늘+1(내일)`.
     - **핵심 불변식**: "오늘 날짜의 활성 버전은 당일에 절대 바뀌지 않는다." 따라서 오늘의 진행률·스트릭은 편집·전환에 의해 당일에 변하지 않는다. 변경은 내일부터 효력을 가진다. (단 최초 활성화는 보호 대상이 없어 예외로 오늘부터 적용.)

- **근거**: 루틴이 편집·공유 가능해지는 순간, "완료 기록을 하드코딩 스케줄 대비 저장"한다는 기존 가정이 무너진다. 스트릭은 과거 60일을 거슬러 "그날 계획을 전부 완료했나?"를 묻는데, "그날의 계획"은 그때 활성이던 루틴/버전에 따라 달라진다. 진행률 분모("그 주의 휴식 아닌 항목 수")도 루틴 의존적이다. 스냅샷 없이 루틴을 편집하면 오늘의 편집이 지난주의 스트릭·진행률을 조용히 다시 써버린다. 이것이 문서 전체를 내부 모순으로 만드는 버그다.

- **대안**:
  - (A) 버전 관리 없이 현재 루틴으로 모든 과거를 계산 → 편집 시 과거가 조용히 바뀜(거부).
  - (B) 매일 그날의 루틴 전체를 완료 로그에 통째로 복사 저장 → 저장 폭증·중복(거부).
  - (C) 활성 전환·편집 시점만 timeline에 기록하고, 버전은 불변 객체로 보관 → **채택**.

- **트레이드오프**: 버전·timeline 구조가 데이터 모델을 복잡하게 만들지만, 이것 없이는 편집·공유 기능과 과거 정확성이 양립할 수 없다. 복잡도는 4장에서 최소 구조로 한정한다.

### D9. 주 기준·시간대 (확정)

- **결정**: 주는 **월요일 시작**으로 고정한다. 날짜·"오늘" 판정은 **기기 로컬 타임존** 기준이다. 주 시작 요일 설정화는 v1 범위 밖(향후).
- **근거**: 1인 로컬 앱이므로 기기 타임존이 곧 사용자 기준이다. 월요일 시작은 원본 스펙(`week:<월요일>`)을 승계한다.
- **대안**: 일요일 시작 / UTC 기준 / 사용자 설정.
- **트레이드오프**: 여행 등 타임존 이동 시 "오늘" 경계가 기기 시계를 따라 이동하지만, 1인 로컬 앱에서 이는 자연스러운 동작이다. DST 전환·자정 경계의 정확한 "오늘" 판정 세부는 미해결로 남긴다(10.1 Q4).

## 4. 데이터 모델

### 4.1 설계 원칙

1. **정의와 기록의 분리**: 루틴이 "무엇을 할 계획인가"(정의)와 "실제로 무엇을 했나"(기록)는 별개의 저장 단위다.
2. **버전 불변성(immutability)**: 한 번 만들어진 루틴 버전은 절대 변형되지 않는다. 편집은 항상 새 버전을 만든다(append-only).
3. **활성 타임라인(activation timeline)**: "언제 어떤 루틴의 어떤 버전이 활성이었는가"를 append-only 이벤트 목록으로 기록한다. 이것이 과거 재계산의 핵심 색인이다.
4. **날짜 → 타임라인 → 버전 → 계획**: 임의의 과거 날짜에 대해, 타임라인을 조회해 그날 활성이던 버전을 찾고, 그 버전의 계획으로 진행률·스트릭을 계산한다.
5. **단일 스키마 재사용**: 아래 스키마는 데이터 모델, 공유 페이로드, 스트릭/진행률 예시에서 동일하게 재사용한다. 스키마 간 drift를 금지한다.

### 4.2 엔티티 개요

```
AppState
├── routines: Routine[]            // 라이브러리 (보유한 모든 루틴)
├── activationTimeline: Activation[]  // append-only, 정렬된 활성 이벤트
├── completionLogs: { [date]: DayLog }  // 날짜별 완료 기록
└── settings: Settings

Routine
├── id
├── name
├── createdAt
├── hidden?: boolean               // 표시 보조(선택). true면 라이브러리에서 숨김. 버전 데이터는 보존(4.8)
└── versions: RoutineVersion[]     // append-only, 버전 히스토리

RoutineVersion (불변)
├── versionId
├── createdAt
├── restDays: Weekday[]            // 휴식일 = 데이터 속성 (D8.6)
└── days: { [weekday]: DayPlan }   // 요일별 계획

DayPlan
├── aerobic: ExerciseSlot[]        // 유산소 종목들
└── anaerobic: ExerciseSlot[]      // 무산소 종목들

ExerciseSlot
├── slotId                          // (버전, 요일, 카테고리) 범위에서 안정적인 위치 식별자
├── name
└── sets                            // 세트 정보 (자유 문자열, 예: "4 × 12")

Activation (append-only)
├── effectiveFrom: date            // YYYY-MM-DD (이 날짜부터 적용)
├── routineId
└── versionId

DayLog (완료 기록)
├── date
├── routineId                       // 그날 활성이던 루틴 (캐시/검증용)
├── versionId                       // 그날 활성이던 버전 (핵심 참조)
└── checks: { aerobic: {slotId: bool}, anaerobic: {slotId: bool} }
```

### 4.3 핵심 결정: 활성 타임라인(activation timeline)

과거 재계산을 위해, "switch active routine"과 "edit routine"을 모두 타임라인에 **append-only 이벤트**로 기록한다. 어떤 경우에도 과거 버전이나 과거 타임라인 엔트리를 변형하지 않는다.

- **활성 루틴 전환** → 타임라인에 `{ effectiveFrom: 내일, routineId: 새 루틴, versionId: 그 루틴의 최신 버전 }` 1건 append. (오늘 보호: 오늘의 활성 버전은 당일에 바뀌지 않는다 — D8.8.)
- **루틴 편집** → 그 루틴에 새 `RoutineVersion`을 append하고, 편집된 루틴이 **현재 활성이면** 타임라인에 `{ effectiveFrom: 내일, routineId: 동일, versionId: 새 버전 }` 1건 append. (활성이 아니면 타임라인 변화 없음 — 다음에 활성화될 때 최신 버전이 적용됨.)
- **최초 활성화**(이전 활성 루틴 없음) → `{ effectiveFrom: 오늘, ... }`. 보호할 오늘 기록이 없으므로 당일부터 적용(D8.8).
- **임의 날짜 D의 활성 버전 조회** = 타임라인에서 `effectiveFrom <= D`인 엔트리 중 가장 최근 것의 `versionId`.

이 구조가 세 가지 케이스를 모두 정확히 처리한다(모든 변경은 내일부터 효력 — 오늘 보호 불변식, D8.8):

| 케이스 | 동작 | 오늘 영향 | 과거 영향 |
|--------|------|----------|----------|
| (a) 루틴 전환 | `effectiveFrom: 내일`로 새 엔트리 append | **없음** — 오늘은 전환 전 버전 유지 | 없음 (전환 이전 날짜는 이전 엔트리 참조) |
| (b) 주중 편집 | `effectiveFrom: 내일`로 새 버전 엔트리 append (forward-only) | **없음** — 오늘은 편집 전 버전으로 계산 | **없음** — 편집 이전 날짜는 옛 버전으로 계산 |
| (c) 휴식일 변경 | 새 버전의 `restDays`가 달라짐, `effectiveFrom: 내일` | **없음** — 오늘 휴식 판정은 옛 버전 | 없음 — 과거 날짜는 옛 버전의 `restDays`로 판정 |

> **(b) forward-only 명시**: 주중에 루틴을 편집해도 새 버전은 **내일(편집일+1)부터** 적용된다. 편집한 당일(오늘)을 포함해 그 이전 날짜의 완료는 모두 옛 버전 기준으로 계산되고, 다음날부터만 새 버전 기준이다. 예를 들어 수요일에 편집하면 수·화·월은 옛 버전, 목요일 이후가 새 버전이다. 한 주가 두 버전에 걸칠 수 있으며, 진행률·스트릭은 **날짜별로** 해당 버전을 조회해 계산한다(주 단위로 버전을 뭉뚱그리지 않는다). 이 정책 덕분에 오늘 이미 기록한 진행률·스트릭이 당일 편집으로 흔들리지 않는다(D8.8 오늘 보호 불변식).

> **필드 설명**
> - `schemaVersion`: 앱 데이터 스키마 버전. 마이그레이션과 공유 호환성 판정에 사용(7장).
> - `restDays`: 휴식일은 버전 데이터에 속한다. 캘린더 고정이 아니다(D8.6).
> - `slotId`: **한 `(버전, 요일, 카테고리)` 범위에서 안정적인 위치 식별자**다. 같은 버전 안에서도 요일·카테고리가 다르면 `slotId` 값이 재사용될 수 있다(예: 4.4 예시에서 월·화 모두 `a1`을 쓰지만 서로 다른 운동이다). 완료 체크는 `checks[category][slotId]` 경로로 그날 그 카테고리 안에서 매핑되므로 충돌하지 않는다. 편집해 새 버전이 생겨도 과거 버전의 `(요일, 카테고리, slotId)` 매핑은 그대로 유지된다.
> - `completionLogs[date].versionId`·`routineId`: 그날 활성이던 버전·루틴의 **비정규화 캐시**다. **계획 해소(date → version)의 단일 출처는 활성 타임라인(4.3)이며**, 진행률·스트릭은 항상 타임라인을 통해 `plan(date)`를 조회한다(4.6, 4.7). `DayLog`에 함께 저장하는 이유는 무결성 검증(기록 시점 버전과 타임라인 재유도 결과의 일치 확인)과 디버깅 편의이며, **계산의 해소 경로로 사용하지 않는다**.
>   - **오늘도 캐시와 타임라인이 발산하지 않는다**: 오늘 기록 시 저장된 `versionId`는, 이후 오늘 안에 루틴을 편집·전환해도 그대로 유효하다. `effectiveFrom = 내일` 정책(D8.8)에 따라 오늘의 활성 버전이 당일에 바뀌지 않기 때문이다. 따라서 `plan(today)`로 재유도한 결과와 `DayLog`에 캐시된 `versionId`가 항상 일치한다.
> - **완료 로그가 없는 날도 타임라인으로 계획을 해소한다.** 사용자가 체크하지 않은 비휴식일은 `DayLog`가 없지만, 진행률 분모 산정(4.6)과 스트릭의 휴식일 통과 판정(4.7)을 위해 그날의 계획이 필요하다. 이 계획은 `DayLog`가 아니라 타임라인 조회로 얻는다. 따라서 `DayLog.versionId`를 해소 경로로 쓸 수 없다(없는 날이 존재하므로).

### 4.5 파생값: "카테고리 완료"와 "그날 완료" (일반화)

기존 스펙의 "러닝 완료(단일 boolean)" + "근력 완료(운동 전부 체크)"라는 비대칭을 제거하고, 유산소·무산소를 동일한 규칙으로 통일한다.

```
// 두 리졸버 — 날짜를 타임라인으로 해소한다(4.3). 모든 계산의 단일 진입점.
//   versionOf(date): 그날 활성이던 RoutineVersion (restDays를 가짐). 없으면 null.
//   plan(date)     : 그날의 DayPlan (aerobic / anaerobic 배열을 가짐). 없으면 null.
versionOf(date) = timeline에서 effectiveFrom <= date 인 엔트리 중 최근 것의 RoutineVersion (없으면 null)
plan(date)      = versionOf(date)?.days[weekdayOf(date)]   // DayPlan, 또는 null

// 한 카테고리가 그날 완료됐는가 = 그 카테고리의 모든 슬롯이 체크됨
// log(date)가 없으면(완료 로그 없는 날) 모든 슬롯을 미체크로 간주 → 미완료
categoryDone(date, category) =
  plan(date) != null &&
  plan(date)[category].length > 0 &&
  plan(date)[category].every(slot =>
    log(date)?.checks?.[category]?.[slot.slotId] === true)

// 빈 카테고리(슬롯 0개)는 "해당 없음" — 완료/미완료 판정에서 제외한다.

// 휴식일 판정 = 그날 활성 버전의 restDays에 포함되는가 (캘린더 아님)
// 활성 버전이 없으면(versionOf(date) == null) 휴식일이 아님으로 본다(빈 날은 별도 처리)
isRestDay(date) =
  versionOf(date) != null &&
  versionOf(date).restDays.includes(weekdayOf(date))

// 그날에 토글 가능한 슬롯이 1개라도 있는가
hasAnySlot(date) =
  plan(date) != null &&
  (plan(date).aerobic.length + plan(date).anaerobic.length) > 0

// 그날 전체 완료 = 슬롯이 있는 모든 카테고리가 완료됨
// plan(date) == null(활성 루틴 없는 날) 또는 빈 비휴식일(슬롯 0개)이면 false →
//   스트릭/진행률은 각자 이 두 경우를 "통과/중립"으로 처리한다(아래 4.7, 4.6 가드 참조)
dayComplete(date) =
  plan(date) != null &&
  !isRestDay(date) &&
  hasAnySlot(date) &&
  ['aerobic','anaerobic']
    .filter(c => plan(date)[c].length > 0)
    .every(c => categoryDone(date, c))
```

> 유산소가 단일 종목 1개여도(예: 러닝 1개) 동일하게 "그 슬롯 체크 = 카테고리 완료"로 처리된다. 무산소가 5개 종목이면 5개 전부 체크해야 카테고리 완료다. 규칙이 하나로 통일된다.
>
> **빈 날 처리 규칙 (스트릭·진행률 일치)**: 다음 두 종류의 날은 "완료/미완료를 따지지 않는 중립 날"로, 스트릭에서는 통과(끊지도 더하지도 않음)하고 진행률 분모에서는 제외한다.
> - **활성 루틴 없는 날**(`plan(date) == null`): 앱 사용 이전 과거 등. 타임라인에 해당 날짜 이전 엔트리가 없는 경우.
> - **빈 비휴식일**(`!isRestDay && !hasAnySlot`): 휴식일은 아니지만 그날 배정된 운동이 0개인 경우(예: 토요일에 유산소만 있고 무산소 0개가 아니라, 아예 두 카테고리 모두 0개인 날).
>
> 이 일치 규칙으로 스트릭의 통과 판정(4.7)과 진행률의 분모 제외(4.6)가 같은 집합의 날을 동일하게 다룬다.
>
> **`log(date)?.` null-safe**: 완료 로그가 없는 날은 모든 슬롯을 미체크로 간주한다. 따라서 미기록 날은 `categoryDone == false`, `dayComplete == false`이며, 스트릭에서 미래로 향하던 연속이 그 지점에서 끊긴다(과거 미기록 날에서 break). 단 오늘은 4.7의 `back == 0` 분기로 보호된다.

### 4.6 진행률 계산 (버전 인식)

```
// 한 주(월~일)의 진행률 — 날짜마다 그날의 버전으로 항목을 조회
function weekProgress(weekStartMonday):
  done = 0
  total = 0
  for date in [weekStartMonday .. +6]:
    p = plan(date)                         // 그날 DayPlan (한 주가 두 버전에 걸칠 수 있음)
    if p == null: continue                 // 활성 루틴 없는 날(versionOf == null) → 분모 제외(중립)
    if isRestDay(date): continue          // 휴식일은 분모에서 제외
    // 빈 비휴식일(슬롯 0개)은 아래 카테고리 루프에서 자연히 total 가산이 없으므로 중립
    for category in ['aerobic','anaerobic']:
      if p[category].length == 0: continue // 빈 카테고리 제외
      total += 1                           // 카테고리 단위로 1 항목 (유산소 칩 / 무산소 칩과 1:1)
      if categoryDone(date, category): done += 1
  return { done, total, pct: total > 0 ? round(done / total * 100) : 0 }
```

> **분모 정의**: "그 주의 휴식 아닌 날에, 슬롯이 1개 이상 있는 카테고리의 개수." 기존 스펙의 "토글 가능 항목 수"를 카테고리 단위로 일반화한 것이다. 유산소 칩 1개 + 무산소 칩 1개 = 그날 최대 2 항목. 날짜마다 버전을 조회하므로 주중 편집이 있어도 정확하다. `plan(date) == null`인 날과 빈 비휴식일은 분모에서 제외해 스트릭의 통과 규칙(4.7)과 일치시킨다.
>
> **분모 산식 예시 (홈 개념도 6.4의 "5 / 8" 도출)**: 4.4 예시 루틴(`restDays: ["sun"]`, 유산소 월~토 6일 + 무산소 월·화 2일)을 이번 주 전체에 적용하면, 카테고리 단위 항목 수(분모 `total`)는 다음과 같이 8이 된다.
>
> | 요일 | 유산소 | 무산소 | 그날 항목 수 |
> |------|--------|--------|-------------|
> | 월 | 있음(1) | 있음(1) | 2 |
> | 화 | 있음(1) | 있음(1) | 2 |
> | 수 | 있음(1) | 없음(0) | 1 |
> | 목 | 있음(1) | 없음(0) | 1 |
> | 금 | 있음(1) | 없음(0) | 1 |
> | 토 | 있음(1) | 없음(0) | 1 |
> | 일 | 휴식일 — 제외 | | 0 |
> | **합계(분모 `total`)** | | | **8** |
>
> **핵심은 분모가 "요일 수"가 아니라 "휴식 아닌 날 × 슬롯 있는 카테고리 수"의 합이라는 점이다.** 분자 `done`은 그중 `categoryDone == true`인 항목 수다. 8개 항목 중 5개 완료면 `5 / 8 = 63%`이며, 이것이 6.4 홈 개념도의 표시 수치와 정합한다.

### 4.7 스트릭 계산 (버전 인식)

```
// 오늘부터 과거로 최대 60일 소급
function streak(today):
  count = 0
  for back in 0 .. 60:
    date = today - back days
    if versionOf(date) == null: // 활성 루틴 없는 날 → 통과(끊지도 더하지도 않음)
      continue
    if isRestDay(date):        // 휴식일은 통과 (끊지도 더하지도 않음)
      continue
    if !hasAnySlot(date):      // 빈 비휴식일(슬롯 0개) → 통과 (진행률 분모 제외와 일치, 4.6)
      continue
    complete = dayComplete(date)
    if back == 0 && !complete: // 오늘은 미완료여도 안 깸 (진행 중)
      continue
    if complete:
      count += 1
    else:
      break                    // 슬롯이 있는 비휴식일을 미완료(미기록 포함)하면 연속 종료
  return count
```

> **버전 인식의 핵심**: `isRestDay(date)`는 `versionOf(date)`(그날 활성 RoutineVersion)를, `dayComplete(date)`는 `plan(date)`(그 버전에서 도출한 그날의 DayPlan)를 통해 계산된다. 두 리졸버 모두 타임라인(4.3)으로 날짜를 해소하므로, 루틴을 전환·편집·휴식일 변경해도 과거 각 날짜는 그날의 계획으로 정확히 판정된다. 오늘의 편집이 과거 스트릭을 바꾸지 않는다.
>
> **null/빈 날 가드(4.8 운영화)**: 루프 진입부 세 가드가 4.8 엣지 케이스를 의사코드 레벨로 처리한다. (1) `versionOf(date) == null`(활성 루틴 없는 날) 통과, (2) 휴식일 통과, (3) 빈 비휴식일(슬롯 0개) 통과. 이 세 가드는 진행률 분모 제외(4.6)와 정확히 같은 집합의 날을 다루므로 스트릭과 진행률이 일치한다. (모든 요일이 버전의 `days`에 존재하므로 `versionOf(date) == null`과 `plan(date) == null`은 동치다.)
>
> 이 알고리즘은 원본 스펙 §6 의사코드를 그대로 옮긴 것이 **아니다**. 원본은 하드코딩 스케줄·일요일 고정 휴식을 가정했으나, 여기서는 휴식일과 계획이 모두 버전 데이터에서 나온다.

### 4.8 엣지 케이스 처리

| 케이스 | 처리 |
|--------|------|
| 활성 루틴이 없는 날(타임라인에 해당 날짜 이전 엔트리 없음) | `plan(date) == null`. 진행률·스트릭에서 제외(휴식처럼 통과). 4.5~4.7 의사코드의 `plan(date) == null` 가드로 운영화됨. |
| 완료 로그는 있는데 해당 버전이 삭제됨 | 버전은 append-only로 절대 삭제하지 않음(불변성). 따라서 발생 불가. 라이브러리에서 루틴을 "숨김" 처리해도 버전 데이터는 보존. |
| 빈 비휴식일(슬롯 0개, 휴식일 아님) | `!isRestDay && !hasAnySlot`. 진행률 분모 제외 + 스트릭 통과(4.5 빈 날 처리 규칙). |
| 같은 날 두 번 활성 전환/편집 | 모든 엔트리가 `effectiveFrom = 내일`(D8.8)이므로 **오늘 기록은 보호된다**. 같은 `effectiveFrom`에 다수 엔트리가 생기면 가장 마지막 append가 유효(타임라인 순서 기준). 따라서 오늘의 `DayLog.versionId` 캐시는 오늘 안의 어떤 편집·전환에도 `plan(today)`와 일치를 유지한다. |
| 임포트한 루틴의 `slotId`가 기존과 충돌 | 임포트 시 루틴 `id`·`slotId`를 앱 측에서 재발급(remap)하여 충돌 회피(7장). |

## 5. 기능 명세

각 기능에 수용 기준(acceptance criteria)을 둔다. 수용 기준은 측정 가능한 형태로 기술한다.

### 5.1 일일 체크

- 각 요일 카드에 유산소 칩 + 무산소 칩을 표시한다(그날 계획에 해당 카테고리가 있을 때만).
- 칩 탭 → 완료/미완료 토글 → 즉시 저장(완료 로그 갱신).
- 칩 색은 카테고리로 구분하지 않는다: 유산소·무산소 모두 미완료엔 무채색이고, 카테고리는 칩 라벨 텍스트로 구분한다. **완료 시에만 Toss Blue로 점등**한다(6.1 색 규칙).
- 완료 시 체크 표시 + 칩 점등(파랑).

**수용 기준**
- AC-5.1.1: 칩을 탭하면 사용자 추가 동작 없이 완료 상태가 저장된다.
- AC-5.1.2: 앱을 껐다 켜도 마지막 체크 상태가 유지된다.
- AC-5.1.3: 카테고리에 슬롯이 0개인 요일에는 해당 칩이 표시되지 않는다.
- AC-5.1.4: 완료 로그에는 그날 `versionId`가 함께 저장된다.

### 5.2 무산소(및 유산소) 운동 펼치기

- 칩 옆 펼치기 아이콘으로 운동 목록을 확장/접는다.
- 개별 운동을 체크할 수 있다(세트 정보 함께 표시).
- 운동을 전부 체크하면 그날 해당 카테고리가 자동 완료 처리된다(파생값, 4.5).
- 칩을 직접 탭하면 그 카테고리 운동을 전체 일괄 체크/해제한다.

**수용 기준**
- AC-5.2.1: 운동 1개라도 미체크면 카테고리 칩은 미완료로 표시된다.
- AC-5.2.2: 전부 체크되면 칩이 즉시 완료 상태로 바뀐다(별도 저장 없이 파생).
- AC-5.2.3: 칩 일괄 탭은 그 카테고리 모든 슬롯을 동일 상태로 만든다.
- AC-5.2.4: 유산소·무산소 모두 동일한 펼치기·체크 규칙을 따른다(비대칭 없음).

### 5.3 루틴 에디터 (생성·편집·버전)

- 루틴 이름을 정한다.
- 유산소/무산소 카테고리에 운동을 추가/삭제/순서변경한다(이름 + 세트 정보).
- **기본 운동 카탈로그 제공(v1 필수)**: 운동 추가 시 사전 정의된 기본 운동 목록(러닝·수영·사이클·요가·스쿼트 등)에서 선택해 빠르게 추가한다. 목록에 없는 운동은 직접 이름을 입력해 커스텀으로 추가한다. 카탈로그 항목은 카테고리(유산소/무산소)와 기본 세트 표기 제안값을 가지며, 선택 후 사용자가 세트 문자열을 수정할 수 있다.
- 운동을 요일에 배정한다(요일별 DayPlan 구성).
- 휴식일을 지정한다(`restDays`).
- 저장 시:
  - 새 루틴이면 `Routine` + 첫 `RoutineVersion(v_001)` 생성.
  - 기존 루틴 편집이면 **새 `RoutineVersion` 생성(append)**. 옛 버전은 변형하지 않는다.
  - 편집 대상이 현재 활성이면 타임라인에 `effectiveFrom: 내일` 엔트리 append(4.3, D8.8 오늘 보호). 활성이 아니면 타임라인 변화 없음.
- 편집 결과는 **내일 홈부터 보인다**(오늘 카드는 옛 버전 기준 표시). 오늘 보호 불변식(D8.8)의 의도된 동작이며, 에디터 저장 후 "내일부터 적용됩니다" 안내를 둔다(5.4 전환 당일 표시 규칙과 동일).

**수용 기준**
- AC-5.3.1: 루틴을 편집하면 옛 버전 객체는 바이트 단위로 변하지 않는다(불변성).
- AC-5.3.2: 편집 후 과거 주의 진행률·스트릭 값이 편집 전과 동일하다(G3).
- AC-5.3.3: 주중 편집 시 편집일 **당일까지(오늘 포함)는 옛 버전**, **다음날부터 새 버전**으로 계산된다(4.3-b, D8.8). 편집 직후 오늘의 진행률·스트릭 값이 편집 전과 동일하다.
- AC-5.3.4: 휴식일을 바꿔도 과거 날짜의 휴식 판정은 그날 버전의 `restDays`를 따른다.
- AC-5.3.5: 운동 추가 시 기본 운동 카탈로그 목록에서 항목을 선택하거나, 목록에 없으면 커스텀 이름을 직접 입력해 추가할 수 있다. 카탈로그에서 선택하면 해당 항목의 카테고리와 기본 세트 표기가 채워지고, 추가 전 세트 문자열을 수정할 수 있다.

### 5.4 루틴 라이브러리 (다수 보유·활성 전환)

- 보유한 모든 루틴을 목록으로 표시한다.
- "새 루틴 만들기", "활성으로 설정", "편집", "복제", "숨김/삭제", "공유"를 제공한다.
- 활성 루틴은 1개만. 활성 표시(배지)와 함께 강조한다.
- 활성 전환 시 타임라인에 엔트리 append(`effectiveFrom: 내일`, D8.8 오늘 보호). 단 최초 활성화(이전 활성 루틴 없음)는 `effectiveFrom: 오늘`.
- **전환 당일 홈 표시 규칙**: 전환하면 `settings.activeRoutineId`는 즉시 새 루틴(B)으로 바뀌지만, 오늘의 계획 해소는 `plan(today)` = 이전 루틴(A)이다(타임라인이 단일 출처, D8.8). 따라서 **오늘 홈의 요일 카드·칩·진행률·스트릭은 이전 루틴(A) 기준으로 표시**하고, "내일부터 '(B 이름)'이 적용됩니다" 안내 배너를 둔다. 라이브러리 화면의 활성 배지는 새 루틴(B)에 표시한다(다음 활성이 B임을 나타냄). 이 "선택된 활성(B) vs 오늘 적용 중인 계획(A)" 구분은 Q1과 연결된다.

**수용 기준**
- AC-5.4.1: 임의 시점에 `settings.activeRoutineId` 기준 활성 루틴은 정확히 1개다. (전환 당일에는 이 "선택된 활성"이 B이고 "오늘 적용 중인 계획"은 A일 수 있으며, 둘의 구분은 5.4 표시 규칙·Q1을 따른다.)
- AC-5.4.2: 활성 전환은 **내일부로** 적용되고(최초 활성화 제외) 오늘·과거 주를 바꾸지 않는다(D8.8). 전환 직후 오늘의 진행률·스트릭이 전환 전과 동일하다.
- AC-5.4.3: 루틴 삭제는 버전 데이터를 보존하는 "숨김"으로 동작한다(4.8). 완전 삭제는 과거 로그가 없는 루틴에만 허용한다.

### 5.5 공유·임포트 (코드·딥링크·QR)

- **내보내기**: 선택한 루틴의 **현재(또는 지정) 버전 템플릿만**을 직렬화 → 공유 코드 문자열 / 딥링크 / QR로 생성. 완료 로그는 포함하지 않는다(D8.5).
- **가져오기**: 코드 붙여넣기 / 딥링크 탭 / QR 스캔 → 미리보기 → "라이브러리에 추가".
- 임포트 시 루틴 `id`·`slotId`를 재발급하여 충돌을 회피한다(4.8).
- 페이로드는 `schemaVersion`을 포함하며, 앱은 호환 가능한 버전만 임포트한다(7장).

**수용 기준**
- AC-5.5.1: 공유 페이로드에 완료 로그가 절대 포함되지 않는다.
- AC-5.5.2: 가져온 루틴은 라이브러리에 새 루틴으로 추가되며 기존 루틴/로그에 영향이 없다.
- AC-5.5.3: 비호환 `schemaVersion` 페이로드는 명확한 안내와 함께 임포트가 거부된다.
- AC-5.5.4: 가져온 루틴을 편집하면 내 루틴처럼 새 버전이 생성된다.

### 5.6 주간 진행률 (버전 인식)

- 상단 진행 바: 이번 주(보고 있는 주) 완료 항목 / 전체 항목 (%).
- 분모·분자는 4.6의 버전 인식 계산을 따른다.

**수용 기준**
- AC-5.6.1: 진행률은 날짜별 버전을 조회해 계산한다(주 단위로 버전을 뭉뚱그리지 않음).
- AC-5.6.2: 휴식일과 빈 카테고리는 분모에서 제외된다.

### 5.7 스트릭 (버전 인식)

- 오늘 기준 과거로 거슬러 "그날 계획 전부 완료"한 날을 연속 카운트(4.7).
- 휴식일은 통과, 당일 미완료는 안 깸, 최대 60일 소급.

**수용 기준**
- AC-5.7.1: 스트릭 계산의 휴식·완료 판정은 모두 그날 버전을 사용한다.
- AC-5.7.2: 루틴 편집·전환 후 과거 구간의 스트릭 기여가 변하지 않는다.

### 5.8 주 이동

- 이전/다음 주로 탐색. 미래 주는 이동 불가(이번 주가 상한).
- 주 라벨에 날짜 범위 + "이번 주 / 지난 주" 표기.

**수용 기준**
- AC-5.8.1: 다음 주 버튼은 이번 주에서 비활성화된다.
- AC-5.8.2: 과거 주를 열면 그 주 각 날짜의 버전으로 데이터가 표시된다.

### 5.9 오늘 강조

- 현재 요일 카드에 "오늘" 태그 + 강조. 이번 주를 볼 때만 적용.

**수용 기준**
- AC-5.9.1: 과거 주를 보면 "오늘" 강조가 표시되지 않는다.

### 5.10 초기화

- 버튼 라벨: **"보고 있는 주 초기화"** → 현재 보고 있는 주의 완료 체크 전부 해제(완료 로그에서 그 주 날짜 제거). 5.8에서 과거 주 열람이 가능하므로, 과거 주를 보고 있으면 그 과거 주가 초기화 대상이 된다(라벨과 동작 일치).
- 루틴 정의·버전·타임라인은 건드리지 않는다(완료 기록만 초기화).

**수용 기준**
- AC-5.10.1: 초기화는 보고 있는 주의 완료 로그만 비우고 루틴 정의·버전·타임라인을 변형하지 않는다.
- AC-5.10.2: 초기화 전 확인 단계를 거친다(실수 방지). 과거 주 초기화 시 "과거 주를 초기화합니다"를 확인 문구에 명시한다.

## 7. 공유 아키텍처

### 7.1 v1 — 서버리스 (코드 / 딥링크 / QR)

**페이로드 = 루틴 템플릿만 (완료 로그 제외)**

```jsonc
{
  "schemaVersion": 1,
  "type": "routine-share",
  "routine": {
    "name": "여름 컨디셔닝",
    "version": {
      "restDays": ["sun"],
      "days": {
        "mon": {
          "aerobic":   [{ "name": "러닝 가볍게", "sets": "30분" }],
          "anaerobic": [
            { "name": "푸시업", "sets": "4 × 한계-2" },
            { "name": "턱걸이", "sets": "4세트" }
          ]
        }
        // ... tue ~ sun
      }
    }
  }
}
```

> 페이로드에는 앱 내부 `id`·`slotId`·`completionLogs`·`activationTimeline`을 **담지 않는다**. 받는 쪽이 임포트 시 `id`·`slotId`를 재발급한다. 이로써 페이로드가 작고, 충돌이 없고, 완료 기록 유출이 없다(D8.5).

**직렬화·전달 파이프라인**

1. 페이로드를 JSON 직렬화.
2. 압축(예: deflate/gzip)으로 길이 축소.
3. URL-safe Base64 인코딩.
4. 전달 형태:
   - **공유 코드**: 인코딩 문자열(필요 시 사람이 읽기 쉽게 블록 분할). 길면 사용성이 떨어지므로 압축이 중요.
   - **딥링크**: `workouttracker://import?d=<인코딩문자열>` (앱 설치 시 직접 임포트 화면 진입).
   - **QR**: 딥링크 URL을 QR로 인코딩. 페이로드가 커 QR 용량을 초과하면 코드 문자열 공유로 폴백.

**임포트 절차**

1. 입력(코드/링크/QR) 파싱 → Base64 디코드 → 압축 해제 → JSON 파싱.
2. `schemaVersion` 검증(7.3). 비호환이면 거부.
3. 미리보기 표시(운동·세트·요일·휴식일).
4. "라이브러리에 추가" → 새 `Routine` 생성, `id`·`slotId` 재발급, 첫 `RoutineVersion` 구성.

### 7.2 버전 호환성

- 페이로드와 앱 데이터 모두 `schemaVersion`을 가진다.
- 임포트 시: `payload.schemaVersion <= app.supportedSchemaVersion`이면 허용. 더 높으면 "앱 업데이트 필요" 안내 후 거부.
- 하위 호환: 앱은 과거 `schemaVersion` 페이로드를 마이그레이션 함수로 현재 모델에 맞춘다.

### 7.3 v2 — 백엔드 단계적 승격 (전략)

서버리스가 한계에 도달하는 지점(긴 코드의 불편, 공개 갤러리/검색 수요, 멀티 기기 동기화)에서 경량 백엔드(예: Supabase)로 승격한다.

| 단계 | 추가 기능 | 비고 |
|------|----------|------|
| v2.0 | 공유 링크 단축(서버 저장 후 짧은 URL 발급) | 긴 코드 문제 해결. 페이로드는 동일하게 템플릿만. |
| v2.1 | 공개 루틴 갤러리 + 검색 | 루틴 메타데이터(이름·태그·작성자) 색인. 완료 로그는 여전히 비공개. |
| v2.2 | 멀티 기기 동기화 | 계정 도입. 완료 로그 동기화(엔드투엔드 또는 서버 저장). 충돌 해결 정책 필요. |

> 승격해도 D8(정의/기록 분리, 버전 관리, 페이로드=템플릿만)은 유지된다. 백엔드는 "전달·저장 채널"만 바꾸며 데이터 모델의 뿌리는 동일하다.

## 8. 비기능 요구

### 8.1 저장소 권고 — AsyncStorage (v1 기본)

| 항목 | AsyncStorage | MMKV |
|------|--------------|------|
| Expo Go 동작 | O (기본 지원) | X (custom dev build / EAS 필요) |
| 성능 | 충분(소량 JSON) | 더 빠름(대량·빈번 쓰기) |
| 설정 복잡도 | 낮음 | 중간(네이티브 모듈) |

- **권고**: v1은 **AsyncStorage**. 이유: Expo Go에서 즉시 동작, v1 데이터 규모(루틴 수십 개 + 일별 로그)가 작아 성능 충분, 설정 단순.
- **MMKV 승격 조건**: 일별 로그가 수천 건을 넘거나 빈번한 동기 쓰기 병목이 측정되면 dev build 전환과 함께 MMKV로 교체 검토. 저장 접근을 추상화 레이어(Repository) 뒤에 두어 교체 비용을 낮춘다.

### 8.2 오프라인 우선

- v1 전 기능이 네트워크 없이 동작한다(공유 코드 생성·임포트 포함; QR/딥링크는 기기 간 직접 전달).
- 네트워크 의존은 v2 백엔드 기능에만 한정.

### 8.3 성능

- 홈 화면 진입 시 이번 주 렌더링과 진행률·스트릭 계산이 체감 지연 없이 완료(목표: 메인 스레드 블로킹 16ms/프레임 이내, 스트릭 60일 소급 계산은 단순 루프이므로 부담 없음).
- 칩 체크는 옵티미스틱 UI로 즉시 반영, 저장은 비동기.

### 8.4 데이터 마이그레이션

- 앱 데이터는 `schemaVersion`을 가진다.
- 앱 시작 시 저장된 `schemaVersion`과 현재 버전을 비교, 필요한 마이그레이션 함수를 순차 적용.
- 마이그레이션 전 자동 백업(이전 데이터 보존)으로 실패 시 복구.
- 버전 불변성 덕분에 과거 완료 로그의 `versionId` 참조는 마이그레이션 후에도 유효해야 한다(마이그레이션이 버전 객체 구조를 바꾸면 참조 매핑을 함께 갱신).

### 8.5 데이터 백업/복원 (로컬)

- 설정에서 전체 `AppState`를 JSON으로 내보내기/가져오기(7장 공유와 별개; 이쪽은 완료 로그 포함 전체 백업).
- v1 동기화 부재를 보완하는 수동 백업 경로.
