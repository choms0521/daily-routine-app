# dailyroutine — 프로젝트 규약

개인용 운동 트래커 앱. 사용자가 만든 주간 운동 루틴을 매일 체크하고, 주간 진행률과 연속 달성(스트릭)을 추적하며, 루틴을 코드/링크/QR로 공유한다. React Native + Expo, Toss 라이트 디자인.

---

## 문서 디렉토리 구조 (docs/)

산출물은 역할에 따라 `docs/` 하위 폴더로 나누어 저장한다. **문서를 만들 때는 그 성격에 맞는 폴더를 선택한다.**

| 폴더 | 역할 | 저장 대상 |
|------|------|----------|
| `docs/prd/` | **기획서 (PRD)** | 제품 요구사항 정의서. 무엇을·왜 만드는가. 제품 결정 기록, 기능 명세, 수용 기준, MVP 범위, 미해결 질문. 구현 방법은 다루지 않는다. |
| `docs/spec/` | **큰 그림의 설계서** | 시스템·아키텍처 설계. 데이터 모델, 모듈 구조, 화면/컴포넌트 구조, 상태관리, 공유 프로토콜 등 "어떻게 만드는가"의 전체 그림. PRD를 구현 가능한 설계로 구체화한다. |
| `docs/development/` | **상세 개발계획서** | 구현 직전 수준의 실행 계획. day-by-day work package, 마일스톤별 작업 분해, 테스트 계획, 종료 조건. 누가·언제·무엇을 코딩하는가. |

### 분류 기준

- **무엇을/왜** → `docs/prd/`
- **어떻게(설계의 전체 그림)** → `docs/spec/`
- **언제/어떤 순서로(실행 계획)** → `docs/development/`

흐름: `prd` → `spec` → `development` 순으로 상세화한다. 상위 문서가 단일 출처(single source of truth)이며, 하위 문서는 상위 문서의 결정을 따른다.

### 날짜 폴더 규약 (필수)

각 역할 폴더(`prd`/`spec`/`development`) 밑에 **문서를 작성하는 날의 날짜 폴더(`YYYY-MM-DD`)** 를 만들고(없으면 생성), 그 날짜 폴더 밑에 문서를 저장한다.

- 경로 형태: `docs/<역할>/<YYYY-MM-DD>/<문서>.md` (예: `docs/prd/2026-06-23/workout-tracker-prd.md`, `docs/spec/2026-06-23/00-architecture.md`)
- 같은 날 추가·개정하는 문서는 그날 날짜 폴더에 함께 둔다.
- 문서 간 상호 참조 경로도 날짜 폴더를 포함한 전체 경로(`docs/<역할>/<YYYY-MM-DD>/...`)로 적는다.

### spec과 development의 경계 (판정 기준)

`docs/spec/`(설계서)과 `docs/development/`(개발계획서)는 상세도로 갈린다. **spec은 "무엇을 어떤 구조로 만드는가"(설계의 그림)이고, development는 "그것을 며칠에 걸쳐 어떤 순서로, 어떻게 검증하며 만드는가"(실행 계획)다.** 한 문서가 두 성격을 섞지 않는다.

**development로 보낸다 (spec에 두지 않는다)**

- 구체적 테스트 fixture와 입력 → 기대 출력 수치 (예: `weekProgress → {done:5, total:8, pct:63}`)
- 실행 가능한 명령 형태의 종료 조건 (예: `jest __tests__/domain`, `tsc --noEmit`, `Object.keys(version.days).length === 7`)
- day-by-day work package (Day 1에 무엇, Day 2에 무엇)
- 작업 시간 배분·일정·진행 체크리스트

**spec에 남긴다 (대략적 설계)**

- 모듈/파일 목록과 각 모듈의 역할
- 함수 시그니처와 역할(인터페이스 수준). 함수 내부의 구체 분기 값·매직넘버까지는 내려가지 않고, 필요하면 PRD 의사코드를 인용하는 선에서 그친다.
- 데이터 모델 타입·스키마의 구조
- 화면/컴포넌트 구조, 상태관리 구조, 공유 프로토콜
- "어떤 기능을 어디까지"의 범위(in/out)
- 테스트는 "무엇을 검증할 방향인가"만 한 줄 수준. 구체 케이스 수치·실행 명령·커버리지 목표 측정은 development로 보낸다.

**경계 한 줄 판정**: "이 내용이 없으면 설계를 이해할 수 없는가"면 spec, "이 내용은 구현·검증을 실제로 수행할 때 필요한가"면 development다.

**development 문서 형식**

- 마크다운 day-by-day로 작성한다. 이 프로젝트의 docs는 마크다운을 기본으로 한다. `~/.claude/rules/plan-documents.md`의 HTML 단일 파일 규약은 ralplan 등 합의 워크플로로 도출된 상세 계획서에 한정되며, 본 프로젝트의 일반 개발계획서에는 적용하지 않는다.
- 단계(마일스톤)별 1개 파일: `docs/development/<YYYY-MM-DD>/stage-N-*.md`.
- 권장 구성: 개요(목표·의존·기간) / Day별 work package(목표·산출물·작업·측정 가능한 종료 조건) / 상세 테스트 케이스(fixture·입출력) / 종료 조건(실행 명령).

## 작성 규약

- 문서 본문은 표준 기술 한국어(평이한 존댓말 또는 `-다`체 평서문)로 작성한다. 대화용 페르소나 표현(사극톤, 호칭, 인사 마무리 등)은 문서에 넣지 않는다.
- **한자(중국어 문자) 일절 금지.** 한글 옆 한자 병기도 금지. 순 한글로만 표기한다. 영어는 type 식별자, CLI 명령어, 표준 단축키 등 기술 용어에만 허용한다.
- 합의 워크플로(ralplan 등)로 APPROVE를 통과한 최종 문서는 성격에 맞는 `docs/` 하위 폴더에 저장한다. 작업 중 임시본은 `.omc/`에 두되 APPROVE 직후 `docs/`로 옮기고 `.omc/` 임시본은 정리한다.
- 합의 사이클의 정정 이력 메타(`v1 → v2 deltas`, `// 정정` 주석 등)는 본문에 남기지 않는다. 최종안만 본문에 보존한다.
